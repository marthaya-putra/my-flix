## Goal
When a stage (mainly `finding_titles`, the slow LLM call + fallback chain) takes a while, the single static copy string reads as dead air. Rotate through up to N messages per stage at a 2s cadence so the UI feels alive. **Decoration only** — messages are generic, never claim a fallback is happening (the client can't know that). Only `finding_titles` gets multiple messages today; every other stage stays single-message.

## Design decisions (confirmed with you)
- **Dumb component**: takes `messages: string[]` + `className?`. Owns shimmer span + text + rotation timer. Not aware of stages, `STAGE_*` maps, or fallback semantics.
- **Uniform data shape**: `STAGE_MESSAGES: Record<StreamStage, string[]>` replaces `STAGE_COPY`. Length-1 arrays for all stages except `finding_titles` (3).
- **Server / wire / generator untouched**: this is a client-only presentation change. The `progress` event still carries just the `stage` enum.

### 2. NEW `src/components/recommendations/rotating-message.tsx`
A dumb component — rotation only, no stage awareness:
```tsx
import { useEffect, useState } from "react";

interface RotatingMessageProps {
  messages: string[];
  className?: string;
}

export function RotatingMessage({ messages, className }: RotatingMessageProps) {
  const [idx, setIdx] = useState(0);

  // Reset to first message whenever the message set changes (i.e. stage change).
  // Safe because parents pass identity-stable refs (module constants).
  useEffect(() => setIdx(0), [messages]);

  // Rotate every 2s, holding on the last message.
  useEffect(() => {
    if (messages.length <= 1) return;
    const id = setInterval(() => {
      setIdx((i) => (i >= messages.length - 1 ? i : i + 1));
    }, 2000);
    return () => clearInterval(id);
  }, [messages]);

  return (
    <span
      className={`bg-gradient-to-r from-muted-foreground via-foreground to-muted-foreground bg-[length:200%_100%] bg-clip-text text-transparent animate-[shimmer_2s_linear_infinite] ${className ?? ""}`}
    >
      {messages[idx]}
    </span>
  );
}
```
- Name tentative (`RotatingMessage`) — open to `StageMessage` if you prefer.
- No crossfade between messages — just a text swap under the existing continuous shimmer. (Keeps it dumb; crossfade can be a follow-up.)

## Behavior summary
- 1 message → no timer, static (identical to today for `loading_preferences` / `looking_up_posters` / `finalizing` / fallback).
- `finding_titles` (3 messages): t=0→"Thinking up titles you'll love", t=2s→"Still narrowing things down…", t=4s→holds on "Almost there…" until the stage resolves.
- Stage change (e.g. `finding_titles` → `looking_up_posters`) resets idx to 0 because the `messages` ref identity changes.
- No message count or cadence is hardcoded into the component — it's driven entirely by the array length and a single `2000` constant.

## Out of scope
- No server/wire changes; no fallback-aware (factual) messaging.
- No crossfade/transition animation between messages.
- `STAGE_LABELS` / `computeProgress` (unused "found of target" path) untouched.
- Deduping the two slightly-different `TypingDots` copies — left as-is.
